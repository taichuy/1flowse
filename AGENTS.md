# 文件管理约定
1.理论上来说单个代码文件不应该超过1500行
2.当前一个目录下不文件不应该超过15个，超过后应该收纳整理对应子目录
3.测试文件统一放到对应子目录下的_tests
4.如果对应子目录下有AGENTS.md，需要先介绍阅读再做处理
# 产品设计与协作原则
1.开发前端功能时，涉及新增页面、新流程、交互流、视觉方案，需要先与开发者沟通；开发者应作为体验第一用户，优先判断交互是否方便、界面是否好看。
2.页面中的 AI 协作层能力，尤其是辅助人生成和编排工作流的部分，落地实现前需要先与开发者沟通。
3.API 设计需要对 AI 友好：参数名统一使用驼峰命名，请求字段尽量少，嵌套层级尽量浅，单次接口语义和职责保持单一。
4.如果 URL 参数长度超过 100 个字符，或 JSON 请求体超过 1000 个字符且包含多层嵌套结构，应优先拆分查询和动作，避免要求模型一次性构造过长、过深的参数。

# 架构设计原则
1.单一职责按变化原因判断；文件、接口、组件是否拆分或复用，以是否会引起变化传播为核心判断标准。
2.如果一个文件或实现只有一个地方使用，且没有明确复用价值，就不要为了抽象而拆分；如果已有成熟依赖组件或现成功能可复用，则优先复用。
3.代码功能层级要分明；耦合的本质是变化传播，解耦的目标是切断变化传播路径，对外契约保持稳定，内部实现允许演化。
4.稳定核心用于承载业务规则、状态机和核心模型；业务层负责决定“该不该做”，边界与适配层负责处理“怎么做到”。
5.外部变化不应直接污染核心规则；接口、组件依赖、插件协议以及其他外部接入点的变化，应尽量收敛在边界和适配层。
6.状态设计必须明确状态集合、流转规则和动作约束；关键模型、关键状态、关键业务动作都应有清晰且唯一的管理入口。
7.唯一入口体现为职责单一和统一管理：同一类能力应由同一模块集中承载，并在该层统一注册或对外暴露，例如认证能力集中在认证模块、接口入口集中在 Controller 层。
