# Frontend Communication Gate

## Ask First

下列改动先与人沟通，再实现：

| 变更类型 | 先沟通 | 原因 |
| --- | --- | --- |
| 需求模糊、页面目标或成功标准未定义 | Yes | 不清楚在解什么问题，容易把参考图误当需求 |
| 只给图片、截图、竞品页或外部视觉样本 | Yes | 需要先拆清哪些借结构、哪些不能照搬视觉皮肤 |
| 新页面或页面 recipe 调整 | Yes | 改变任务域边界和页面语法 |
| 新流程 | Yes | 改变任务路径和预期 |
| 交互流调整 | Yes | 改变用户行为契约 |
| L1 模型调整 | Yes | 会影响 Drawer / Inspector 统一规则 |
| 状态语义调整 | Yes | 会影响全站运行态一致性 |
| 视觉方案调整或引入外部视觉母本 | Yes | 容易打碎统一视觉语言 |
| 页面内 AI 协作层能力 | Yes | 属于产品级行为，不只是界面实现 |

## Usually Safe To Do Directly

下列改动通常可直接按 Skill 执行：

| 变更类型 | 通常可直接做 | 前提 |
| --- | --- | --- |
| 局部 bugfix | Yes | 不改变既有页面 recipe 和交互契约 |
| 同类页面内复用已有组件 | Yes | 不引入新交互规则 |
| 职责收敛 | Yes | 只是把混杂实现拆清楚 |
| 明显重复代码整理 | Yes | 没有过早抽象 |
| 在既有视觉基线内补齐缺失状态或响应式 | Yes | 不改动语义映射和页面边界 |

## Escalate Immediately

出现以下信号时，不要继续闷头写：

- 同类对象要采用不同点击行为
- 需要新增第三种 L1 详情模型
- 想把 `awesome-design-md` 或别的视觉样本直接当成当前项目规范
- `overview` 开始承接完整编排、完整日志、API 正文或说明性文案
- 页面逻辑里开始出现大量协议转换、拼接和条件分发
- 你已经说不清“这段状态到底归页面、组件还是共享层”

## Clarify Before Implementing

当需求模糊、只有图片或用户说“照这个做一个页面”时，先做下面三件事：

1. 明确页面目标、目标用户、核心动作和成功标准。
2. 说清参考图里哪些只借结构 / 信息层级 / 节奏，哪些视觉语言必须回到 `DESIGN.md`。
3. 先输出设计需求草案和集中待确认问题，等人确认后再实现。
