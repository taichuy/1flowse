# Frontend Requirement Refinement

## When To Enter

命中下面任一信号，就不要直接实现：

- 用户只说“做个页面 / 优化一下 / 改成这种感觉”
- 用户只给图片、截图、竞品页、Dribbble / Awwwards 链接
- 用户想借外部样本，但没有说明借结构、借视觉还是借交互
- 这是新页面、新流程、新视觉方向，但页面目标和成功标准还没落地

## Goal

把模糊请求收敛成一份可确认的前端设计需求，再进入实现。

图片和外部样本只能提供灵感、结构和节奏，不能直接替代当前项目的 `DESIGN.md`、页面 recipe 和状态语义。

## Hard Rules

- 新页面、新流程、外部视觉母本、图片驱动需求，未确认前不实现。
- 参考图必须拆成“可借”和“不可借”两部分，不能默认整份照搬。
- 视觉语言、状态颜色、组件边界最终以 `DESIGN.md` 和现有页面 recipe 为准。
- 如果问题本质是信息架构、层级、入口或交互直觉，补用 `frontend-logic-design`。
- 与用户沟通时，优先一次性集中给出待确认问题，不要把决策问题拆成多轮零散追问。

## What To Extract

先从请求里提炼这几类信息：

1. 页面目标
   - 这个页面服务谁
   - 用户进来后最重要的动作是什么
   - 什么结果算完成

2. 页面骨架
   - 是列表页、详情页、工作台、设置页、配置页，还是混合页
   - 顶部、侧栏、主区、辅助区分别承载什么
   - 哪些模块是必须项，哪些只是参考图里的装饰项

3. 关键对象与状态
   - 页面主对象是什么
   - 需要哪些状态：`loading / empty / error / success / running / selected / draft`
   - 哪些状态是当前项目已有语义，哪些只是参考图表现

4. 参考图借鉴边界
   - 借布局骨架
   - 借信息层级
   - 借卡片/列表/工具条的组合方式
   - 不直接借品牌色、文案、图标和第三方产品气质

## Default Output Format

在实现前，先向用户输出一个设计需求草案，推荐固定结构：

1. 任务理解
   - 我理解你要做的页面是什么，用来解决什么问题。

2. 参考图拆解
   - 这张图里我建议借哪些东西。
   - 这些东西不能直接照搬，必须回到当前项目规范。

3. 设计需求草案
   - 页面目标
   - 主要用户
   - 页面结构
   - 关键模块
   - 关键动作
   - 关键状态
   - 视觉约束

4. 待确认问题
   - 一次性列出会影响方向的决策问题，让用户统一回答。

5. 明确建议
   - 给出默认推荐方向，并说明为什么。

## Suggested Questions

优先确认这些问题：

- 页面名称和主要对象是什么
- 用户最常做的 1 到 2 个动作是什么
- 参考图里你最想继承的是结构、视觉，还是交互节奏
- 哪些模块是必须有的，哪些只是“像那个图一样有感觉”
- 这个页面需要覆盖哪些状态和异常场景
- 视觉上必须继承 `DESIGN.md` 的哪些规则，哪些允许局部借鉴外部样本

## Example Framing

如果用户只丢一张图，可以先这样收敛：

> 我会把这张图当成结构和信息节奏参考，不直接复刻它的品牌皮肤。先按当前项目的 `DESIGN.md` 回收视觉语言，再整理一版页面目标、模块划分和待确认问题给你，确认后再开始实现。
