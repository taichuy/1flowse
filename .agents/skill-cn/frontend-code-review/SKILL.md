# 前端代码审查

## 意图

只要用户要求审查前端代码（特别是 `.tsx`、`.ts` 或 `.js` 文件），就使用此技能。支持两种审查模式：

1. **待定更改审查** – 检查计划提交的暂存/工作树文件，并在提交前标记违反检查清单的行为。
2. **以文件为目标的审查** – 审查用户指定的特定文件，并报告相关的检查清单发现。

对每个适用的文件和模式，都要遵守下面的检查清单。

## 检查清单

参见 [references/code-quality.md](references/code-quality.md)、[references/performance.md](references/performance.md)、[references/business-logic.md](references/business-logic.md) 获取按类别划分的动态检查清单——将其视为要遵循的规范规则集。

用紧急元数据标记每个违规行为，以便未来的审查者可以优先修复。

## 审查流程

1. 打开相关的组件/模块。收集与类名、React Flow Hook、属性记忆化和样式相关的行。
2. 对于审查点中的每个规则，记录代码偏离的地方并捕获代表性的片段。
3. 按照下面的模板撰写审查部分。首先按 **Urgent**（紧急）标志分组违规行为，然后按类别顺序（代码质量、性能、业务逻辑）分组。

## 要求的输出

当调用时，响应必须严格遵循以下两个模板之一：

### 模板 A (有发现)

```
# 代码审查
发现 <N> 个紧急问题需要修复：

## 1 <bug 的简要描述>
文件路径: <path> 行 <line>
<相关代码片段或指针>


### 建议的修复
<建议修复的简要描述>

---
... (重复每个紧急问题) ...

发现 <M> 个改进建议：

## 1 <建议的简要描述>
文件路径: <path> 行 <line>
<相关代码片段或指针>


### 建议的修复
<建议修复的简要描述>

---

... (重复每个建议) ...
```

如果没有紧急问题，省略该部分。如果没有建议，省略该部分。

如果问题数量超过 10 个，总结为“10+ 个紧急问题”或“10+ 个建议”，并仅输出前 10 个问题。

不要压缩部分之间的空行；保持原样以提高可读性。

如果您使用模板 A（即有需要修复的问题），并且至少有一个问题需要更改代码，请在结构化输出后附加一个简短的后续问题，询问用户是否要应用建议的修复。例如：“您希望我使用建议的修复部分来解决这些问题吗？”

### 模板 B (无问题)

```
## 代码审查
未发现问题。
```
