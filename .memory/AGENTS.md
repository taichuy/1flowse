# 背景

`.memory` 是开发者私有记忆目录，用来记录 AI 与开发者之间的约定、协作边界和可复用决策。

进入用户交互后，优先阅读本文件；如果记忆仍为空，与用户沟通初始化。

所有日期统一使用绝对时间格式：`yyyy-mm-dd hh`。

## 检索规则

1. 固定先读 `.memory/AGENTS.md`。
2. 固定再读 `.memory/user-memory.md`。
3. 对 `feedback-memory`、`project-memory`、`reference-memory`、`tool-memory`，第一轮只读取每个文件前 30 行的 `YAML front matter`。
4. `.memory/project-memory/archive/` 仅在当前有效记忆缺失、当前正文不足或需要追溯同主题旧阶段时读取。
5. 单轮最多扫描 200 个记忆文件。
6. 单轮只展开与当前任务最相关的最多 5 条有效记忆全文。
7. 有效记忆选择原则固定为：宁缺毋滥，只选有用的，可少选，不凑满 5 条。
8. 是否直接参考某条记忆，不只看时间，还要结合记忆类型、`decision_policy` 与当前任务相关性判断。
9. `project-memory` 超过两天后，仅在其确实影响当前决策时，才回到当前代码、当前文档或当前运行结果验证。
10. `reference-memory` 只作为索引，不直接作为结论来源。
11. `tool-memory` 只在当前任务会用到对应工具或该工具刚刚失败时参与检索。
12. 检索到足够支撑当前判断的有效证据后停止；不要为了措辞、全面感或凑满配额继续展开记忆。
13. 如果记忆与最新 AGENTS、README、代码或运行结果冲突，以最新可验证来源为准，并在必要时更新记忆。

## 记忆存储规则

记忆的目标不是做用户画像，而是让 AI 在后续任务中更快、更准地复用已有结论。

### 存储成功标准

每轮最终回复前必须静默判断是否需要维护记忆。

成功标准：

1. 应存的长期偏好、反馈纠正、阶段共识、外部入口或工具失败解法已写入对应记忆。
2. 已有同主题记忆可承载时优先更新旧文件，不重复新建。
3. 不应存的一次性流水账、代码可读事实、临时任务摘要没有沉淀。
4. 没有命中存储触发器时，直接结束，不需要向用户解释。

### 存储触发器

命中以下任一条件时，必须先维护记忆，再最终回复。

#### 用户记忆

用户表达长期稳定偏好、协作方式、技术背景、决策习惯或沟通风格时，更新 `.memory/user-memory.md`。

典型话术：`以后都`、`我更喜欢`、`不要再`、`默认按`、`我习惯`。

#### 反馈记忆

用户对 AI 的做法给出纠正、否定、认可或边界澄清时，写入 `feedback-memory`。

典型话术：`你这里不应该`、`这样不对`、`这个方向是对的`、`以后遇到这种情况`、`不是 X，而是 Y`。

反馈记忆应遵守以下规则：

1. 每一条都包含：规则、原因、适用场景。
2. 既记录纠正，也记录肯定，避免 agent 越来越保守。
3. 按 `interaction/` 和 `repository/` 两类分目录维护。
4. 在 `YAML front matter` 中必须显式写 `feedback_category`，用于说明是哪一类纠正。
5. 使用 `YAML front matter` 作为摘要层，`decision_policy` 默认使用 `direct_reference`。

#### 项目记忆

用户确认阶段性方案、架构方向、当前任务边界、短期状态或已批准计划时，写入 `project-memory`。

项目记忆正文至少回答：

1. 谁在做什么？
2. 为什么这样做？
3. 为什么要做？
4. 截止日期？
5. 决策背后动机？

项目记忆使用 `YAML front matter` 作为摘要层，`decision_policy` 默认使用 `verify_before_decision`。

#### 引用记忆

用户提供仓库外资料、外部路径、参考项目、网页或文档入口，且后续可能复用时，写入 `reference-memory`。

引用记忆只记录“去哪里看什么”的入口索引，不记录正文结论。

引用记忆使用 `YAML front matter` 作为摘要层，`decision_policy` 默认使用 `index_only`。

#### 工具记忆

项目环境中发生真实工具失败，且本轮验证出可复用解决办法时，写入 `tool-memory`。

边界如下：

1. 只记录真实失败过的问题。
2. 只记录已验证可复用的解法。
3. 不写通用工具使用文档。
4. 不写未来风险提示。
5. 同一工具、同一问题、同一处理办法复现时，直接追加到原文件。

工具记忆使用 `YAML front matter` 作为摘要层，`decision_policy` 默认使用 `reference_on_failure`。

### 写入预算与停止条件

1. 单轮最多新增或更新 3 条记忆。
2. 长期用户偏好优先级高于单次项目事实，反馈记忆优先级高于项目记忆。
3. 只记录后续会复用的信息；普通任务完成摘要不写记忆。
4. 某条信息 30 天后仍然有效时，优先沉淀为用户记忆、反馈记忆或稳定规则；否则保持为项目记忆。
5. 完成必要写入后停止，不为措辞、全面感或归档洁癖继续扩写。

## 不存什么

过多无关记忆会显著增加检索噪声，以下内容不应放入记忆中：

1. 可直接从当前代码中读取的代码模式、架构和路径。
2. 版本管理历史，优先回看 `git`。
3. 不能复用的一次性调试过程或一次性修复流水账。
4. 配置文件中已经明确声明的内容。
5. 临时性任务细节。
6. 与记忆目录无关的运行摘要、草稿或历史杂项文件。
7. 已经被 AGENTS、README、skill 或代码稳定承载的规则复述。

## 目录索引

- `.memory/user-memory.md`
  - 用户记忆主文件，固定入口。
- `.memory/feedback-memory/`
  - 反馈记忆根目录，存放模板与分类子目录。
- `.memory/feedback-memory/interaction/`
  - 用户沟通、执行流程、记忆检索等交互类纠正。
- `.memory/feedback-memory/repository/`
  - 仓库结构、目录管理、脚本放置、版本控制等工程类纠正。
- `.memory/project-memory/`
  - 当前有效的项目记忆目录，存放阶段事实、短期共识和模板。
- `.memory/project-memory/archive/`
  - 已归档的旧阶段 project-memory；仅在需要追溯同主题历史阶段时读取。
- `.memory/reference-memory/`
  - 引用记忆目录，存放入口索引和分类说明。
- `.memory/tool-memory/`
  - 工具记忆目录，按工具分目录、按问题建文件，记录真实失败案例与解法。
- `.memory/todolist/`
  - AI 自主开发且用户离线时的待确认目录，处理后直接删除，不做沉淀。
