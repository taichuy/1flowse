# Plugin Ecosystem Architecture

## 背景

当前文档里已经明确 7Flows 首版会兼容 Dify 插件生态，但如果直接把 Dify 兼容写死在 API 后端里，会带来几个长期问题：

- 平台内部会逐渐被某一个外部生态的协议和目录结构牵着走
- 未来若要发展 7Flows 原生插件生态，边界会变得模糊
- 未来若要再接第二个生态，例如 n8n，就会不断往核心后端叠加特判

这不符合 `7Flows IR` 优先和“兼容层不反向主导内部设计”的原则。

## 目标

把插件相关架构明确为：

- 7Flows 原生插件生态是主线
- Dify 兼容层只是一个可启用/可停用的兼容层插件/服务
- 平台允许未来存在多个兼容层并存

## 设计理念提炼

这次补充不只是为了回答“Dify 该怎么接”，更是为了固定 7Flows 后续处理外部生态时的一组判断原则：

### 1. 先保内部事实，再谈外部兼容

- `7Flows IR`、运行时事件流、授权模型、发布网关是平台主体
- 外部生态只能映射进这些内部事实，不能反过来决定内部模型边界

### 2. 兼容层应是叶子，不应长成树干

- 兼容层应该挂在平台边缘，通过统一插件接口或独立服务接入
- 不应让 Dify 兼容逻辑渗透进核心执行链、核心对象模型和产品主导航

### 3. 冷启动复用不等于长期依附

- 首版兼容 Dify 是为了缩短能力冷启动周期
- 这不代表 7Flows 要把自己的插件体系、节点体系和产品结构永久做成 Dify 的镜像

### 4. 先分层，再扩生态

- 先区分 `node`、`provider`、`compat_adapter`
- 再分别设计注册、发现、调用、鉴权、生命周期和 UI 展示
- 否则后面只会得到一个越来越混乱的“插件大列表”

### 5. 多生态并存要从第一天就可表达

- 即便当前只接 Dify，也要从命名、注册、部署和管理视图层面支持 `native / compat:*`
- 否则未来每接入一个生态，就要回头重写一遍平台边界

### 6. 可启停比“内建默认存在”更符合平台化

- 兼容层应能独立启停、限工作空间、限插件类型、做健康检查
- 这样平台主体保持干净，也更适合自托管和后续企业部署

## 决策

### 1. 插件体系分层

建议平台内统一区分三类插件：

- `node`
  - 直接为工作流提供节点能力
- `provider`
  - 提供模型或外部服务供应商能力
- `compat_adapter`
  - 提供某个外部插件生态到 7Flows 的桥接能力

其中：

- `node` 和 `provider` 是 7Flows 原生生态的主体
- `compat_adapter` 是为冷启动复用外部生态服务

### 2. 原生后端与兼容层服务的边界

推荐边界如下：

- `api/`
  - 维护 7Flows 原生插件注册中心
  - 定义统一插件发现、调用、鉴权、生命周期协议
  - 对运行时暴露统一的工具/供应商能力
- `compatibility adapter service`
  - 每个外部生态一个独立服务或独立部署单元
  - 负责该生态的安装、运行、调试、健康检查与协议转译

首版可直接按下面理解：

- 7Flows API 是平台主体
- Dify Adapter Service 是一个单独的外部生态代理

### 3. Dify 兼容层不是硬编码功能

Dify 兼容应被建模为：

- `compat:dify` 生态
- 可启用/可停用
- 可独立观察健康状态
- 可独立安装/卸载插件

不应建模为：

- API 中永远存在的一套硬编码兼容逻辑
- 平台唯一插件生态

### 4. 多生态并存的产品表达

插件管理和后续仓库组织都应按生态分类，而不是“一个插件列表包打天下”：

- `native`
- `compat:dify`
- `compat:n8n`（未来预留）

这会影响：

- 插件发现接口
- 插件管理页
- 运行时能力来源标记
- 安装源和生命周期管理

### 5. 为什么这比“直接把 Dify 兼容做进后端”更好

- 更符合 `7Flows IR` 优先
- 兼容层可独立演进，不污染核心运行时
- 原生生态和外部兼容生态能清楚分层
- 后续新增第二个生态时不需要重做核心架构
- 也更利于部署端按需开启，减少无关依赖

## 与 Dify 本地文档的关系

本次理念整理参考了本地 Dify 文档：

- `E:\code\taichuCode\dify-docs\zh\develop-plugin\dev-guides-and-walkthroughs\cheatsheet.mdx`

从该文档可见，Dify 当前插件类型至少包括：

- 工具插件
- 模型插件
- 智能体策略插件
- 扩展插件
- 数据源插件
- 触发器插件

因此 7Flows 在兼容 Dify 时，更适合先以“生态兼容层”承接，而不是直接把这些类型一股脑烙进 7Flows 核心对象模型。

## 下一步建议

后续实现 Dify 插件兼容时，建议优先落这些基础对象：

1. `PluginRegistry`
   - 统一管理 native / compat:* 插件索引
2. `CompatibilityAdapterRegistration`
   - 管理兼容层开关、健康状态与作用域
3. `PluginCallProxy`
   - 先按 ecosystem 路由，再进入具体 adapter
4. `Dify Adapter Service`
   - 作为第一套外部生态兼容服务单独实现
