## 1. 文档目标

本文档用于收口模型供应商“实例选择”专题的第一轮设计，明确：

- 为什么当前“自动选最新 ready 实例”不符合分层目标
- 为什么节点仍应只绑定供应商语义，而不绑定实例标识
- 为什么这一轮只实现供应商级手动主实例，不一起实现自动熔断
- 供应商路由层的职责、数据结构、接口契约和前端交互
- 后续自动熔断、自动切换应如何在这层继续扩展

## 2. 背景与问题

当前系统已经具备：

- 工作区级模型供应商安装与实例管理
- 每个实例独立配置 `enabled_model_ids`
- `agent-flow` 节点通过 `provider_code + model_id` 选择模型供应商

但当前“供应商 -> 实例”的解析规则仍是隐式推导：

- `settings model provider options` 按同一 `provider_code` 下“最新 ready 实例”产出可用模型选项
- 运行时编译上下文也按同一规则推导 `effective_instance_id`
- 编译结果会把该实例标识直接写进 LLM runtime，执行时不再重新解析

这带来三个问题：

1. 业务语义错误
   - 节点明明绑定的是供应商语义，但实际生效实例却被“最新更新时间”偷换。
2. 行为不可控
   - 新增、编辑、刷新任意实例后，都可能意外改变真实调用实例。
3. 无法平滑演进
   - 后续自动熔断、自动切换需要一个稳定的供应商级路由层；如果继续依赖“列表排序即路由规则”，后续会再次返工。

## 3. 设计结论

### 3.1 核心结论

本轮新增“供应商路由层”，固定作用域为 `workspace + provider_code`。

这层只回答一个问题：

- 当前工作区下，这个供应商应解析到哪个实例

节点契约保持不变：

- 节点继续只保存 `provider_code + model_id`
- 节点不保存 `provider_instance_id`

### 3.2 本轮采用的路由模式

本轮只实现一个路由模式：

- `manual_primary`

其含义是：

- 每个 `workspace + provider_code` 最多配置一个主实例
- `settings model provider options` 只暴露主实例的已启用模型
- 运行时编译上下文只解析主实例
- 如果主实例失效，不自动切换到其他实例

### 3.3 本轮不做的内容

本轮明确不做：

- 自动熔断
- 自动切换到下一个实例
- 候选实例池和优先级顺序管理
- 按模型路由
- 健康探测、冷却时间、半开恢复
- 基于错误类别的自动转移策略

## 4. 为什么自动熔断不适合这一轮一起做

自动熔断不是设置页增强，而是运行时路由策略升级。它至少需要同时定义：

- 哪些错误会触发切换
  - 例如 `timeout`、网络错误、`429`、`5xx`
- 哪些错误绝不能切换
  - 例如 `401/403`、模型不存在、配置缺失、参数不合法
- 何时恢复
  - 立即恢复、冷却恢复、半开探活
- 最多切几次
  - 单次请求内允许几次重试和几次实例转移
- 如何保证模型兼容
  - 目标实例也必须支持当前 `model_id`

如果这些规则没有一起冻结，直接上自动切换只会把真实配置问题掩盖成“偶尔换了一个实例也能跑”，排查成本会更高。

因此本轮正确边界是：

- 先把“供应商级手动主实例”做成稳定真值层
- 后续自动熔断在这层扩展，不再回头改节点契约

## 5. 分层与职责

### 5.1 节点层

节点负责：

- 声明自己需要哪个 `provider_code`
- 声明自己要用哪个 `model_id`

节点不负责：

- 选择具体实例
- 保存实例标识
- 管理供应商级故障转移策略

### 5.2 供应商路由层

供应商路由层负责：

- 保存每个供应商当前主实例
- 向设置页和 `agent-flow` 提供统一“当前生效实例”真值
- 向运行时编译提供稳定解析结果
- 在未来承载自动熔断、候选池和恢复策略

### 5.3 实例层

实例层继续负责：

- 保存密钥和配置
- 保存该实例的 `configured_models`
- 保存该实例的 `enabled_model_ids`
- 保存该实例的模型缓存与刷新状态

实例层不直接决定：

- 在供应商级语义下，当前应该选中哪一个实例

## 6. 数据模型

### 6.1 新增供应商路由配置表

新增工作区级供应商路由配置表，建议语义如下：

| 字段 | 含义 |
| --- | --- |
| `workspace_id` | 工作区 |
| `provider_code` | 供应商代码 |
| `routing_mode` | 首轮固定为 `manual_primary` |
| `primary_instance_id` | 当前主实例 |
| `created_by` | 创建人 |
| `updated_by` | 更新人 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

约束：

- `workspace_id + provider_code` 唯一
- `primary_instance_id` 必须属于同一 `workspace_id + provider_code`

### 6.2 读模型补充字段

`list model provider instances` 的返回中新增：

- `is_primary: boolean`

用途：

- 设置页直接渲染“主实例”标记
- 避免前端自己再做二次推断

## 7. 后端契约

### 7.1 新增写接口

新增显式写接口：

- `PUT /api/console/model-providers/providers/:provider_code/routing`

首轮请求体固定为：

```json
{
  "routing_mode": "manual_primary",
  "primary_instance_id": "uuid"
}
```

首轮不支持空路由模式切换，也不支持候选池配置。

### 7.2 路由写入校验

写入时必须校验：

- `provider_code` 在当前工作区存在实例
- `primary_instance_id` 存在
- `primary_instance_id` 属于当前工作区
- `primary_instance_id` 的 `provider_code` 与路径参数一致
- `primary_instance_id` 当前状态为 `ready`

### 7.3 删除行为

如果被删除实例正好是当前主实例：

- 自动删除该供应商路由配置
- 该供应商随后变为“未配置主实例”

不做自动补位，不做自动选择“下一个实例”。

## 8. 读路径改造

### 8.1 设置页供应商选项

`listConsoleModelProviderOptions` 改为：

- 先读取 `workspace + provider_code` 的路由配置
- 只取 `primary_instance_id`
- 只暴露该主实例的 `enabled_model_ids`

保留现有响应字段：

- `effective_instance_id`
- `effective_instance_display_name`

但字段来源从“最新 ready 实例”改为“路由配置中的主实例”。

### 8.2 运行时编译上下文

运行时编译上下文改为：

- 不再按 `updated_at` 选实例
- 对每个 `provider_code` 只解析路由配置中的主实例
- 用该实例生成 `effective_instance_id`、`available_models` 和 `allow_custom_models`

### 8.3 无主实例时的行为

当某供应商未配置主实例时：

- 该供应商不进入 `agent-flow` 的模型供应商选项
- 运行时编译解析不到该供应商，返回正式不可用错误

这样可以保持“供应商可用性”与“路由配置完整性”一致，不再做隐式兜底。

## 9. 失效与错误行为

### 9.1 主实例失效

若主实例后续从 `ready` 变成 `invalid` 或 `disabled`：

- 保留路由配置
- 该供应商在设置页显示“主实例不可用”
- `agent-flow` 侧视为该供应商不可用
- 不自动切到其他实例

### 9.2 配置错误不静默漂移

本轮明确禁止以下行为：

- 主实例失效后自动切到最新实例
- 主实例删除后自动选剩余实例中的一个
- 因刷新模型或编辑配置导致主实例隐式漂移

原因是：

- 这些都属于“自动路由策略”，应该放到后续熔断专题中统一建模

## 10. 前端交互

### 10.1 入口位置

入口继续放在现有“供应商实例”弹窗，不下沉到节点层。

原因：

- 这是供应商级能力，不是节点级能力
- 当前弹窗已经按供应商聚合实例，是最自然的路由管理入口

### 10.2 弹窗交互

弹窗顶部新增紧凑的“主实例”选择器：

- 只列出当前供应商 `ready` 状态的实例
- 选中后立即提交路由更新
- 成功后同步刷新实例列表和供应商选项

实例列表中新增：

- 当前主实例的 `主实例` 标记

保留现有操作：

- 编辑 API Key
- 刷新候选模型
- 刷新模型
- 删除实例

### 10.3 总览摘要

供应商总览区域补一个供应商级摘要：

- `主实例：<display_name>`
- 若未配置则显示 `未设置`

这样用户无需每次进入弹窗才能知道当前实际路由目标。

## 11. 对后续自动熔断的扩展点

本轮虽然不实现自动熔断，但要为下轮保留稳定扩展位。

后续专题可以在当前路由层继续扩：

- `routing_mode = failover`
- `candidate_instance_ids`
- `failure_policy`
- `cooldown_seconds`
- `max_handoffs_per_request`
- `health_probe_mode`

扩展原则固定为：

- 节点契约继续不变
- 供应商路由层演进
- 实例层继续只管理配置、模型与健康事实

## 12. 验证要求

本轮实现后至少需要覆盖以下验证：

1. 设置页能显式设置主实例，并正确渲染 `is_primary`
2. `model provider options` 只暴露主实例的启用模型
3. 主实例切换后，`agent-flow` 模型选项同步切换
4. 删除主实例后，供应商变为未配置主实例
5. 主实例失效后，不自动切换到其他实例
6. 运行时编译不再按 `updated_at` 推导实例

## 13. 实施边界总结

这一轮的真实目标不是“让实例更聪明”，而是先把供应商级实例选择从隐式排序规则升级为显式路由真值。

冻结边界如下：

- 做：供应商路由层
- 做：`manual_primary`
- 做：统一设置页、模型选项、运行时编译的实例来源
- 不做：自动熔断
- 不做：自动切换
- 不做：候选池
- 不做：健康恢复策略

这样落地后，系统会先具备：

- 清晰的分层
- 稳定的实例真值来源
- 可继续扩展自动熔断的正确落点

随后下一步才是基于这层继续写 implementation plan，并在单独专题里推进自动熔断。
