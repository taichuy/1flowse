## 1. 文档目标

本文档用于冻结本轮已经确认的前端质量门禁方向：

1. 样式边界检测不以静态 selector 扫描为主，而以真实浏览器渲染回归为主。
2. 检测粒度定位在“中等测试”：
   - 比单纯 lint 更强，能看到真实渲染结果和样式来源。
   - 比全链路 E2E 更轻，不依赖权限、后端和完整业务流都准备完毕。
3. 后续开发完成后，应能针对单组件、页面和文件影响面做快速样式回归，而不是只能手工点页面。

本轮目标不是重写 `web/app` 业务功能，也不是立刻引入完整视觉 diff 平台，而是为后续 QA 与覆盖检测建立一个稳定、可执行的真实渲染入口。

## 2. 当前已确认现状

### 2.1 规则层已经基本存在

- `DESIGN.md` 已有“前端样式边界与 UI 质量门禁”章节。
- `frontend-development` 已写入 `theme token -> first-party wrapper -> explicit slot -> stop` 的实现层顺序。
- `qa-evaluation` 与 `frontend-quality-gates.md` 已写入前端质量 fail 规则。

当前主要缺口不是“有没有规则”，而是“规则如何在开发后被真实执行和回归”。

### 2.2 纯静态扫描不满足本轮需求

用户已明确否决“只扫描写法”的方案，原因成立：

- 样式不一定集中写在独立 `css` 文件，也可能直接位于组件或共享样式模块中。
- 纯扫描只能抓“看起来危险的写法”，不能证明真实渲染结果是否破坏了组件。
- 未来接入权限、后端和更复杂页面后，如果只能走业务路由开组件，测试会被迫退化成重型 E2E。

因此，本轮检测必须先看浏览器中的真实结果，再追踪样式来源，而不是只看源码写法。

### 2.3 全局样式需要收束，但不能被整文件豁免

- `global.css` 这类文件本来就承载全局样式入口，不能因为在全局文件里就天然判错。
- 但也不能因为文件名叫“global”就整文件白名单放行。
- 更合理的判断方式是：
  - 允许全局入口存在。
  - 通过场景回归验证它有没有影响受保护组件。
  - 通过浏览器回溯结果判断当前节点最终样式来自哪里。

## 3. 本轮已确认决策

### 3.1 范围

- 不改 `web/app` 业务行为。
- 本轮新增前端样式边界运行时回归基础设施。
- 脚本入口位于 `scripts/node`，作用目标为 `web/` 前端工作区。
- 继续复用现有 `frontend-quality-gates.md`，不再额外创建重复 checklist 文件。

### 3.2 测试模式

脚本需支持以下运行模式：

1. `component`
   - 指定组件场景。
   - 用独立宿主渲染组件，不依赖业务路由。
2. `page`
   - 指定页面场景。
   - 默认也走独立宿主。
3. `file`
   - 输入文件路径。
   - 通过场景注册表映射到受影响场景后执行。
4. `all-pages`
   - 跑全部页面级场景。

后续允许扩展更多组合模式，但本轮不以命令行花样为目标，而以稳定回归能力为目标。

### 3.3 执行引擎

- 本轮只做真实浏览器渲染。
- 使用无头 Chromium 类浏览器执行。
- 检测顺序固定为：
  1. 渲染目标场景
  2. 定位关键节点
  3. 获取 `computed style`
  4. 回溯命中的 CSS 规则来源
  5. 根据边界规则给出结论

### 3.4 不做中心化豁免白名单

本轮不引入通用 exemption/allowlist 机制。

原因：

- 中心化白名单极易退化成垃圾桶。
- 当前仓库尚未出现一个经过验证、不得不用例外机制才能解决的真实场景。
- 在没有真实样本前先造例外系统，只会弱化门禁。

如后续确实出现无法避免且合理的特殊场景，再基于真实案例单独设计、单独约束，不在本轮预埋通道。

## 4. 推荐架构

### 4.1 总体结构

本轮采用“独立宿主 + 场景注册表 + 浏览器回归脚本”的混合方案：

- `web/app` 中新增一个专用于样式边界回归的独立测试入口。
- 页面和组件默认都通过该入口挂载，不依赖产品导航或业务路由。
- 少量必须验证壳层组合真相的页面，可声明为真实路由场景。
- 根脚本负责启动/复用宿主、驱动浏览器、采集结果并输出报告。

这样可以同时满足：

- 单组件可稳定打开。
- 页面可稳定回归。
- 后续接权限和后端后，不会把样式测试强行推高成完整 E2E。

### 4.2 宿主入口

建议在 `web/app` 内增加独立入口，例如：

- `web/app/style-boundary.html`
- `web/app/src/style-boundary/main.tsx`
- `web/app/src/style-boundary/registry.ts`

该入口不进入产品导航，不承担业务功能，只承担以下职责：

- 根据 URL 或查询参数加载指定场景
- 给场景提供统一的 Provider、主题、容器尺寸和 mock 环境
- 暴露可被浏览器脚本读取的场景元数据

### 4.3 场景注册表

场景注册表是本轮的核心事实源，负责声明：

- 场景 ID
- 场景类型：`component` / `page` / `route`
- 入口组件或页面
- 所需 Provider、mock 数据、初始 props、容器尺寸
- 需要验证的关键节点与样式断言
- 关联文件列表，用于 `--file` 模式映射影响面

约束：

- `--file` 模式不做猜测式扫描。
- 如果文件未声明任何受影响场景，应直接报“覆盖关系未声明”。
- 场景由前端开发自行维护，作为样式回归覆盖面的显式契约。

### 4.4 页面场景策略

- 页面场景默认也走宿主模式，而不是直接走真实业务路由。
- 只有少量必须验证壳层、导航、真实 route 组合关系的页面，才额外声明为真实路由场景。

原因：

- 默认宿主模式更稳定，依赖更少。
- 真实路由场景保留给确实需要验证组合真相的页面。
- 这样能避免“为了开一个页面先把权限、接口、状态全串起来”的测试膨胀。

## 5. 运行时检测模型

### 5.1 断言对象

脚本不直接断言“某个 selector 写得好不好”，而是断言：

1. 目标节点的真实样式是否符合预期。
2. 这些样式最终来自哪个文件和 selector。
3. 如果来源涉及第三方组件内部样式链，是否越过已定义的样式边界。

### 5.2 关键节点定义

每个场景需声明关键节点，例如：

- 菜单项
- 子菜单标题
- 按钮主体
- 卡片头部
- 输入框容器
- 表格表头单元格

节点定位建议基于：

- `data-testid`
- 语义 role
- 稳定类名

不建议依赖脆弱的自动生成类名。

### 5.3 样式断言

每个关键节点至少应声明：

- 需要检查的样式属性
- 期望值或允许区间
- 是否属于保护性断言

保护性断言示例：

- 菜单项 `display` 仍为原生块级布局
- 菜单项 `height` 与 `line-height` 的关系未被破坏
- 子菜单弹层未因共享样式而被改成错误布局

本轮优先覆盖最容易被共享样式打坏的布局属性：

- `display`
- `position`
- `height`
- `min-height`
- `line-height`
- `padding`
- `gap`
- `overflow`

### 5.4 样式来源追踪

脚本需要在浏览器里把命中的样式来源追到：

- 来源文件
- 对应 selector
- 是否属于全局样式、组件样式或第三方默认样式

判断逻辑不是“全局来源一定错”，而是：

- 当前节点的关键属性是否被不合理地来自共享规则覆盖
- 来源 selector 是否越过 `theme token / first-party wrapper / explicit slot / stop` 的边界规则

### 5.5 全局样式判定

`global.css` 在本轮中属于合法来源类型，但不属于天然安全来源。

判定方式：

- 允许它作为命中来源出现。
- 若它改变了受保护节点的关键属性，则需要结合 selector 层级判断是否合法。
- 若它引发真实渲染回归失败，即使文件位于全局入口，也应判定失败。

## 6. 报告与产出

### 6.1 失败报告

脚本失败时至少输出：

- 场景 ID
- 场景类型
- 失败节点
- 失败样式属性
- 实际值与期望值
- 最终命中的文件与 selector
- 失败原因归类

### 6.2 证据产出

当场景失败时，脚本应产出可供 QA 和开发直接使用的证据：

- 截图，统一写入 `uploads/`
- 必要的 DOM 片段或节点摘要
- 样式来源列表

这样后续 QA 不需要只凭“代码看起来像有问题”来下结论。

## 8. 不在本轮范围内

以下内容明确不在本轮范围：

- 完整视觉像素 diff 平台
- 全链路登录、鉴权、后端依赖联动
- 自动推断所有文件与所有场景的影响关系
- 中心化例外白名单系统
- 把样式边界测试并入现有业务路由或产品导航

## 9. 验收标准

本轮设计对应的实现完成后，应满足：

1. 开发者可针对单组件运行真实浏览器样式回归，而不依赖业务路由。
2. 开发者可针对单页面运行真实浏览器样式回归，且页面默认通过独立宿主打开。
3. 开发者可针对单文件触发受影响场景回归；若映射缺失，脚本明确报错而不是假装通过。
4. 脚本失败时可指出关键样式属性、真实值、来源文件与 selector。
5. 全局样式可被追踪和验证，但不会因为位于全局入口就被整文件放行。
6. QA 后续可直接把该脚本输出作为前端质量门禁证据之一。

## 10. 当前结论摘要

- 样式边界检测本轮不走“静态扫描主导”，而走“运行时真实渲染主导”。
- 目标是中等粒度回归：比 lint 强，比完整 E2E 轻。
- 页面和组件都默认走独立宿主；真实路由场景只保留给少量确需组合验证的页面。
- `--file` 模式依赖显式场景注册表，不做拍脑袋影响面推断。
- `global.css` 允许存在，但必须通过真实场景回归来验证影响，不做整文件豁免。
- 本轮不引入中心化白名单式例外机制。
