# ADR-0002：分级 AGENTS、链式说明与本地时序记忆

- 状态：Accepted
- 日期：2026-03-20

## 背景

随着协作文档持续膨胀，根 `AGENTS.md` 同时承载了项目定位、架构边界、团队协作、本地记忆入口、技能索引和大量重复说明，已经开始带来三个问题：

1. 入口文档虽然没有越过 200 行太多，但信息密度过高，目录级约束和仓库级约束混在一起，AI 与人都更难快速定位真正需要的规则。
2. `README.md` 与对外定位文档混入了商业分层、路线判断和重复叙述，偏离了“开源项目入口只说明当前项目能做什么”的目标。
3. `docs/.private/runtime-foundation.md` 缺少统一时间戳和目标账本结构，AI 很难稳定判断当前用户目标是否发生偏移，也不利于把本地连续性记忆做成时序可追溯资产。

## 决策

1. 根 `AGENTS.md` 只保留仓库级最小规则、上下文链和绝对边界，并保持短小可索引。
2. 在 `docs/`、`api/`、`web/`、`services/`、`.agents/` 下新增目录级 `AGENTS.md`，把局部阅读顺序和边界下沉到对应目录。
3. 共享文档默认采用链式说明：入口文档给索引和边界，稳定规则写入 `docs/dev/team-conventions.md`，长期决策写入 `docs/adr/`，专项流程写入 `.agents/skills/`。
4. 对外入口文档与开源项目定位文档只描述当前开源项目定位、能力和诚实边界，不再承载商业分层或详细路线计划；授权细节仍以 `LICENSE` 为准。
5. 技能目录按“元流程 / 后端 / 前端 / 插件服务 / 特例”分组维护，并补充统一索引与使用案例。
6. `docs/.private/runtime-foundation.md` 必须带 `YYYY-MM-DD HH:mm` 时间戳、当前用户目标账本、目标匹配结论和按时间标记的下一步规划。
7. `docs/.private/` 允许初始化为当前开发者自己的本地 git 仓库，用作时序记忆；该仓库不配置共享远端，也不改变其“非共享事实源”属性。
8. durable change 继续要求验证、非交互式 Git 提交，并默认尝试把当前分支推送到远端；如推送失败，需要在最终汇报中说明。

## 后果

- 共享入口会更轻，目录规则和服务规则能在更靠近改动的位置被读取。
- 共享文档里的重复说明和个人机器信息会明显减少，AI 上下文压力更小。
- 本地记忆会更适合做“目标比对 + 时间线追踪”，而不是继续堆一份长篇静态说明。
- 技能体系会更贴近真实目录和服务边界，尤其补齐 `services/` 下的插件 / 兼容服务协作入口。

## 后续动作

- 继续把重复或漂移的规则从共享入口文档迁移到 `team-conventions`、ADR 或 skill。
- 如果未来新增重要一级目录，应同步补目录级 `AGENTS.md`。
- 若插件 / 兼容服务继续扩张，再按服务边界细分更多 skill，而不是重新把规则堆回根文档。
